# T048 — Periodic upload bursts alongside continuous download

## 2026-07-20 — First Attempt (Cold Build)

### What was done
Implemented periodic upload bursts on a separate cellular connection to measure upload throughput without interrupting the continuous download stream:

1. **PeriodicUploadBurster** (`Sources/Live/PeriodicUploadBurster.swift`):
   - New class that runs periodic upload bursts on a separate `.cellular` NetworkConnection
   - Bursts run every ~3 seconds (a named constant for easy tuning)
   - Each burst sends HTTP POST request + ~256KB body, times the transfer, yields a ThroughputSample
   - Uses injected clock (`now` closure) for testability
   - Independent async loop that runs alongside download stream

2. **CellularThroughputSampler** modifications:
   - Now spawns TWO concurrent tasks: continuous download + periodic upload bursts
   - Merges samples from both streams into a single AsyncStream
   - Both tasks are cancelled together when the stream terminates

3. **SimulatedProviders** refactoring:
   - Split `SimulatedSampler` into `runDownloadStream()` and `runPeriodicUploadBursts()` tasks
   - Upload bursts follow the same ~3s cadence as the live PeriodicUploadBurster
   - Continuous download stream unaffected—bursts run concurrently

4. **Unit tests** (`Tests/PeriodicUploadBurstTests.swift`):
   - `testSimulatedBurstsEmitOnIntervalAlongsideContinuousDownloads()`: verifies uploads and downloads both emit, with downloads >> uploads
   - `testUploadSamplesUpdateSessionReading()`: confirms upload samples feed the SweepSession window and update uploadMbps reading
   - `testUploadBurstsAreMinorFractionOfTotal()`: quantifies that uploads stay <25% of session data (satisfies "minor fraction" requirement)
   - `testCancellingStopsDownloadsAndUploads()`: confirms both streams stop when task is cancelled

### Design decisions

- **3-second interval**: Balances responsiveness with data efficiency. ~256KB upload every 3s ≈ 85KB/min or ~5.1 MB/hour, easily minor fraction with 30–60 MB/min download.
- **Separate connection**: Prevents any network congestion or multiplexing issues from the upload burst disturbing the continuous download stream.
- **Concurrent tasks**: Both run as independent async loops, cancellation is synchronized via the stream's termination handler.
- **Simulated cadence**: The fake sampler uses the same ~3s interval so tests exercise realistic upload/download ratios without real network I/O.

### Device verification deferred

The live `PeriodicUploadBurster` uses real `.cellular` NetworkConnection which only works on a real device. Unit tests exercise the pure sampling machinery (upload math, windowing, session state updates) via the fake sampler with injected clock. Manual device verification (T025-style: watch upload readout update periodically while download flows) is required for the live path but outside CI scope.

### Notes

- The upload burst stream and download stream operate entirely independently; burst latency/failure doesn't impact download.
- If a single burst fails (network timeout, etc.), the burster catches the error and continues to the next scheduled burst; no crashes or stalling.
- The interval is exposed as a named constant (`PeriodicUploadBurster.burstInterval`) for future tuning based on empirical battery/data-use targets.
