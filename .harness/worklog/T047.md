## 2026-07-20 — Implementation complete

### What was done
Investigated wiring `SweepSession` (`Sources/Domain/SweepSession.swift`) up to the continuous
sampler (T045) and wall-clock window (T046). Found that `SweepSession`'s existing per-sample
`record(_:)` / tick-driven `republishFromWindow()` / gap-driven `evaluateStaleness()` design (built
in T039–T041, before continuous sampling existed) already satisfies all three "Do" requirements
because it operates purely on whatever samples arrive, regardless of discrete-probe vs. continuous
shape:

1. `record(_:)` already records every incoming `ThroughputSample` into `downloadWindow` and
   accumulates `sessionDownloadBytes` — unconditional on sample cadence.
2. `republishFromWindow()`, driven by the existing 250ms `tickTask` (`ScoutMotion.tickInterval`),
   already recomputes `downloadMbps` from `downloadWindow.megabitsPerSecond(at: now())` using the
   injected clock, independent of sample arrival — so it updates smoothly between chunks.
3. `evaluateStaleness()` already keys off `lastSampleAt`, which `record(_:)` refreshes on every
   sample — so with a continuous stream (chunks arriving well inside the 900ms staleness
   threshold), staleness never spuriously fires between reads; it only fires once chunks genuinely
   stop landing for the threshold duration, exactly the "dead spot" signal the task wants.

No `Sources/Domain/**` changes were needed — the wiring was already correct and generalizes to
T045's continuous per-chunk sampler and T046's wall-clock window without modification.
