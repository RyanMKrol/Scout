# T037 — Short recency-weighted throughput window (live responsiveness)

**Built:** 2026-07-17 21:26 UTC

## Changes Made

1. **Sources/Domain/ThroughputWindow.swift**: Added a named constant
   `ThroughputWindow.liveWindow: Duration = .milliseconds(600)` (marked `nonisolated` — the project
   infers default-actor isolation and a plain `static let` was flagged as main-actor-isolated in a
   `nonisolated init` default). Changed the init's default `window` parameter from `.seconds(2)` to
   `ThroughputWindow.liveWindow`. No change to the transfer-time-weighted math or the
   zero/negative-byteCount / zero/negative-duration drop guards.
2. **Sources/Domain/SweepSession.swift**: Replaced both `ThroughputWindow(window: .seconds(2))`
   construction sites (the stored-property initializers at the top of the class and the
   re-construction inside `subscribeToSamples()`) with `ThroughputWindow(window:
   ThroughputWindow.liveWindow)`. No `.seconds(2)` window remains for the live download/upload reading.
3. **Tests/ThroughputWindowTests.swift**: Added two tests driven by the injected `ContinuousClock`
   style already used in this file (no bare `.now`):
   - `testLiveWindowEvictsSampleOlderThanShortWindow` — a sample older than `liveWindow` (600ms) is
     evicted; only the fresh sample contributes to the reported Mbps.
   - `testLiveWindowConvergesToNewThroughputWithinOneWindowSpan` — after a step change from ~1 Mbps
     to ~100 Mbps, sampling for slightly more than one `liveWindow` span (800ms) after the step
     converges the reported Mbps to within 5 of the new rate (vs. the previous 2s window, which
     would still be diluted by the old samples at that point).

Ran test-first: added the two tests against the still-2s-windowed code first and confirmed
`testLiveWindowConvergesToNewThroughputWithinOneWindowSpan` failed (reported ~34 Mbps, not within
5 of 100) before making the window-duration change, then re-ran to green after the fix.
