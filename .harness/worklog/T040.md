# T040 — Wire up the 250ms display tick

## 2026-07-17

Did:
- Added a repeating main-actor tick to `SweepSession` at `ScoutMotion.tickInterval` (0.25s,
  referenced from `Sources/DesignSystem/ScoutTheme.swift` — no longer dead), started in `start()`
  alongside the existing radio/path/staleness tasks and torn down in `stop()`.
- The tick calls `republishFromWindow()`, which recomputes `downloadWindow`/`uploadWindow`
  `megabitsPerSecond(at:)` from the session's injected clock and republishes `downloadMbps` /
  `uploadMbps` with the same caps `record(_:)` applies (`ScoutMeter.downloadCapMbps` /
  `uploadCapMbps`), so the displayed value eases as samples age out of the short T037 window
  between probes instead of only jumping once per probe lands.
- Guarded the tick with `isMeasuring, cellularAvailable, !isStalled` — once `evaluateStaleness()`
  (T039) has taken over fading the reading to zero past the staleness threshold, the tick backs
  off so the two don't fight over the published value; a fresh sample clears `isStalled` and hands
  control back to the tick.
- Refactored `start()`'s task-spinup into small `startRadioTask()` / `startPathTask()` /
  `startPollingTask(interval:work:)` helpers to keep it under SwiftLint's cyclomatic-complexity
  limit after adding the fourth background task.
- Marked `ScoutMotion.tickInterval` `nonisolated` (it's a plain `Double` constant) — the app
  target's default `SWIFT_DEFAULT_ACTOR_ISOLATION: MainActor` otherwise made it MainActor-isolated,
  which can't be referenced from `SweepSession`'s `nonisolated static let tickInterval: Duration`.
