## 2026-07-17 — download-biased, tighter probing + honest data-budget copy

**What was done**

- `Sources/Live/ProbePacer.swift`: `interval` reduced from `.milliseconds(500)` to
  `.milliseconds(250)` (still a single named constant; `delayBeforeNextProbe` unchanged).
- `Sources/Live/CellularThroughputSampler.swift`: added `ProbeSchedule.direction(forProbeIndex:)`,
  a download-biased sequence (3 download probes per 1 upload probe — every 4th probe is upload).
  The sampler's run loop now derives each probe's direction from a monotonically increasing
  `probeIndex` instead of strictly alternating; download/upload probe bodies themselves are
  unchanged.
- Recomputed the honest data-budget figure: `probeBytes` (262,144 B ≈ 0.262 MB) × up to 4
  probes/sec (240/min) ≈ 63 MB/min at the paced ceiling, down to roughly half that when transfers
  take longer than the 250ms floor — stated as **"30–60 MB per minute"**. Replaced every
  "10–40 MB per minute" occurrence in `Sources/Screens/ConsentView.swift` (the bold consent bullet)
  and `README.md` (5 occurrences: overview bullet, "How it works" §2, trade-offs section, Core UX
  section, roadmap line) with the new figure. Confirmed `grep -rn "10–40 MB" Sources README.md`
  returns nothing.

**Tests**

- Added `Tests/ProbeScheduleTests.swift` (new test file, always in-scope): pins the exact 4-probe
  cycle (`[.download, .download, .download, .upload]`), the 3:1 download:upload ratio over 40
  probes (30 download / 10 upload), and that upload lands on probe indices `3, 7, 11, …`. Against
  the old strict-alternation logic this assertion (`downloadCount == uploadCount * 3`) would fail
  (old logic yields a 1:1 ratio); it now passes against `ProbeSchedule.direction(forProbeIndex:)`.
- Updated `Tests/HTTPProbeRequestTests.swift`'s pacer tests, which hardcoded the old 500ms
  interval: `testDelayBeforeNextProbeAt200MillisecondsIs300MillisecondsRemaining` renamed/rescaled

  remaining of a 250ms interval), `testDelayBeforeNextProbeAtOrPastIntervalIsZero` rescaled to a
  250ms elapsed offset, and added `testIntervalIsAtMost250Milliseconds` to directly pin
  `ProbePacer.interval <= .milliseconds(250)`.
