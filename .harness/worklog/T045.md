# T045 Worklog

## 2026-07-20 — Implementation complete

### What was done
Replaced the paced, discrete 256 KB request/response probe download path (T038's `ProbeSchedule`
3:1 schedule + `ProbePacer`-gated `runDownloadProbe`) with ONE continuous, effectively-unbounded
download stream, per the "live speedometer" model:

1. `Sources/Live/HTTPProbeRequest.swift` — added `downloadBytes` (1 GiB), used only by
   `download()`'s `GET /__down?bytes=…` request. `probeBytes` (262144) is kept as-is (unused by the
   live sampler now, but harmless — a future upload task can reuse or replace it).
2. `Sources/Live/CellularThroughputSampler.swift` — rewritten around a single
   `runContinuousDownloadLoop`: opens one forced-`.cellular` `NetworkConnection`, sends one
   `HTTPProbeRequest.download()`, then loops on `connection.receive(atLeast: 1, atMost: 262144)`
   forever. Each `HTTPResponseParser.Event.bodyProgress` chunk yields a `ThroughputSample` with
   `byteCount` = that chunk's size and `endedAt` = the wall-clock instant it arrived;
   `transferDuration` is now just the delta since the previous chunk (no longer the probe RTT — the
   new rolling window in T046 ignores it per the design note). If the body completes
   (`.responseComplete`, since even a 1 GiB body is finite), the loop transparently re-sends another
   `download()` on the same connection and resets the parser, so the stream never really stops. On a
   connection-level error the loop rebuilds the connection and retries after a 1s backoff, same as
   before. Removed `ProbeSchedule` (the 3:1 download/upload schedule) and all `ProbePacer` usage from
   the download path entirely — upload is out of scope here (T048) and the live sampler currently
   emits download-only samples.
3. `Sources/Domain/SimulatedProviders.swift` — `SimulatedSampler` now emits download chunk samples
   every 40ms (vs. the old 250ms per-probe cadence), each carrying a positive `byteCount` scaled from
   the current walked Mbps value over that 40ms slice, so the scripted fake matches the live
   sampler's continuous-chunk shape instead of one-sample-per-probe. The download-rate random walk
   itself still only advances every 6th tick (~240ms, matching the old walk pace) — only the chunk
   *emission* cadence changed. Upload sampling is kept (also every 6th tick) purely so the existing
   upload gauge / `sessionUploadBytes` wiring in `SweepSession` still has something to display in the
   simulator scenarios; T048 will redesign upload properly.

### Tests
- `Tests/HTTPProbeRequestTests.swift::testDownloadRequestIsByteExact` — updated to assert the new
  1 GiB `bytes=1073741824` request line (was 262144).
- `Tests/ProbeScheduleTests.swift` — deleted; it existed solely to pin the removed 3:1
  `ProbeSchedule.direction` behaviour, which no longer exists.
- `Tests/SimulatedProvidersTests.swift::testGreatScenarioEmitsContinuousDownloadChunkStream`
  (replaces `testGreatScenarioEmitsAlternatingSamples`) — collects 24 samples from the `.great`
  scenario fake and asserts: (a) download samples outnumber what a per-probe 1:1 alternating cadence
  could produce (`> collected.count / 2`, i.e. materially more download chunks than a discrete-probe
  shape would ever yield), (b) every download sample has `byteCount > 0`, and (c) download samples'
  `endedAt` is monotonically non-decreasing across the stream — pinning exactly the "Done when"
  acceptance bar (many incremental chunks, not one-per-probe, positive bytes, advancing wall clock).
  Ran this test FIRST against the old alternating-sample fake (confirmed it failed — collected 4
  download + 4 upload samples with `downloadSamples.count == collected.count / 2`, not `>`), then
  implemented the continuous-chunk fake until it passed.
