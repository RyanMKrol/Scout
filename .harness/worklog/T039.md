# T039 — Staleness decay: dead spots read as dead

## 2026-07-17

**What I did**

- `Sources/Domain/SweepSession.swift`: added a staleness rule. `SweepSession` now takes an
  injected `now: @Sendable () -> ContinuousClock.Instant` clock seam (defaults to a live
  `ContinuousClock()` so the existing `AppEnvironment.swift` call site — out of scope for this
  task — keeps compiling unchanged). Two named constants: `stalenessThreshold` (900ms) and
  `stalenessDecayDuration` (1000ms). A lightweight polling task (`stalenessTask`, started in
  `start()`) periodically calls `evaluateStaleness()`, which compares the injected clock's `now()`
  against the timestamp of the last recorded sample (`lastSampleAt`, also stamped from the
  injected clock, independent of `sample.endedAt`). Once elapsed time exceeds the threshold,
  `isStalled` flips true and `downloadMbps`/`uploadMbps` linearly fade toward 0 over the decay
  window (fully zero once elapsed ≥ threshold + decay). Any fresh sample immediately clears
  `isStalled` and resets the decay baseline. Cellular-unavailable and `stop()` paths also reset
  the stalled state.
- `Sources/Screens/MeasuringView.swift`: when `session.isStalled`, the quality word (existing
  `measuring.quality` identifier) reads "Measuring…" in a muted color instead of a
  confidently-colored quality label, and the status-row pulsing dot also mutes to gray — so a
  frozen-looking healthy number can no longer be mistaken for a live reading.
- `Tests/SweepSessionTests.swift`: added a `ManualClock` test double (injected via the new `now`
  seam) and `testStalledSampleDecaysReadingAndClearsOnFreshSample`, which feeds a sample, advances
  the clock past threshold + decay with no further sample, and asserts `isStalled` becomes true
  and `downloadMbps` decays fully to 0, then asserts a fresh sample clears `isStalled` and
  restores the reading. (The old `SweepSession` had no `isStalled`/`now` API at all, so this test
  could not even compile against the pre-fix code — the implementation and the test seam were
  necessarily added together.)
