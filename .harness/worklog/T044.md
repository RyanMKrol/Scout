## 2026-07-18 — Implementation complete

### What was done
Re-attempt of T042, now re-scoped to include `Sources/Domain/**` + `Tests/**`. Added a tap-to-pause/resume gesture over the central dial in the live measuring screen:

1. `Sources/Screens/MeasuringView.swift`:
   - Extracted the live dial into `liveDial()`, wrapped it with `.contentShape(Circle())` + `.onTapGesture { session.togglePause() }`.
   - Gave the dial's accessibility element the identifier `dial.pauseToggle`, `.isButton` trait, a paused-aware label, and an `.accessibilityAction` so VoiceOver users can pause/resume without a physical tap.
   - Added `pausedDialIndicator()` (pause glyph + "PAUSED" text) shown as an overlay when `session.isPaused`, with a **new** identifier `dial.paused` — merged into a single accessibility element via `.accessibilityElement(children: .combine)` (SwiftUI otherwise exposes each child — Image, Text — as its own separately-identified node, which isn't queryable as one `dial.paused` element).
   - `statusRow` now takes a 3-way `StatusRowState` (`.sweeping` / `.livePaused` / `.emptyPaused`) instead of a bool: the live screen's row keeps its existing identifier `measuring.status` but now also renders a "PAUSED" state when `session.isPaused`; the empty-state row is unchanged (`paused.status`, still only reachable from `pausedContent()`).
   - The tap gesture and both paused identifiers only exist inside `measuringContent()` — `pausedContent()` (no-consent / no-cellular empty states) is untouched, so `paused.status` / `paused.title` / `paused.startButton` keep resolving exactly as before.
   - Footer's `accessibilityValue` now passes `isPaused: session.isPaused` through to `AccessibilitySummary`.
2. `Sources/Domain/AccessibilitySummary.swift`: added an `isPaused: Bool = false` parameter; when true the value is prefixed with "Paused, reading frozen. " ahead of the existing download/upload/generation/quality/session-data summary. Default `false` keeps every existing call site's behavior unchanged.
3. The pause/resume domain logic itself (`SweepSession.pause()/resume()/togglePause()`, freezing the readout) already shipped in T041 — this task is purely the UI wiring.

### Tests
`Tests/MeasuringPauseToggleTests.swift` (new file):
- `testTogglePauseFlipsIsPausedThenFlipsBackOnSecondTap` — pins the tap-to-toggle wiring at the domain layer (`session.togglePause()`, the exact call the dial's tap gesture makes): first call pauses, second resumes.
- `testToggleIsInertWhenSessionIsNotMeasuring` — the dial's tap gesture only exists in `measuringContent()`, gated on the same `consentGiven && cellularAvailable` precondition under which the session runs; pins that a toggle call before `start()` never enables pause (proxy for "the toggle is inactive in the empty state" since the empty-state view never invokes it in the first place).
- `testAccessibilityValueIsPausedAwareWhenPaused` / `testAccessibilityValueDefaultsToNotPaused` — pin `AccessibilitySummary`'s new paused-aware value and its default-false backward compatibility.

**Why a unit-test fallback instead of a committed XCUITest:** the task spec's primary ask was an `XCUITest ... in Tests/`, but `Tests/**` is the `ScoutTests` **unit-test** bundle target (`project.yml`); the real `ScoutUITests` (XCUITest) target sources from `UITests/`, which is **not** in this task's scope (`Sources/Screens/**`, `Sources/DesignSystem/**`, `Sources/Domain/**`, `Tests/**`). Per the task's own documented fallback bar ("If … an XCUITest still cannot be driven headlessly in this harness … the fallback bar is a plain XCTest unit test …"), I used the unit-test fallback rather than touching out-of-scope `UITests/**`.
