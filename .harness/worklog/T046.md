# T046 Worklog

## 2026-07-20 — Implementation complete

### What was done
Replaced `ThroughputWindow`'s transfer-time-weighted rate math (T037) with a true wall-clock
sliding window, per the design note in T045/T046:

- `Sources/Domain/ThroughputWindow.swift` — `Sample` no longer stores `transferDuration` (the
  parameter is still accepted by `record(...)` for call-site compatibility but is unused in the
  rate). `megabitsPerSecond(at:)` now computes `(Σ byteCount in window × 8) / divisorSeconds /
  1_000_000`, where `divisorSeconds` is `min(windowSeconds, max(elapsed-since-oldest-sample,
  minimumDivisorSeconds))` — `minimumDivisorSeconds` (0.05s) is a small positive floor so a
  just-landed single sample doesn't blow up toward infinity. `liveWindow` changed from 600ms to the
  spec'd 500ms named constant. `record(...)` still drops `byteCount <= 0` samples; the
  `transferDuration <= .zero` guard was removed since duration is now vestigial.
- `Tests/ThroughputWindowTests.swift` — rewritten for wall-clock semantics: steady arrival at a
  fixed rate reports ≈ B/Δt regardless of per-chunk transfer duration, samples older than 500ms are
  evicted, the rate goes to nil once arrivals stop and the window elapses, the default `liveWindow`
  is asserted as 500ms, and 1,000,000 bytes over a ~0.49s warmed window reads ≈16 Mbps per the spec
  example.
- `Sources/Domain/SweepSession.swift` was not touched (its `ThroughputWindow` usage is
  signature-compatible) but two of its existing tests encoded values from the old
  duration-weighted math and needed updating for the new wall-clock formula:
  `testPauseFreezesReadingAndResumeRestoresLiveUpdating` and
  `testTickRepublishesFromWindowBetweenProbesAndStopsAfterStop` in `Tests/SweepSessionTests.swift`
  — recomputed expected Mbps values by hand for the new formula (including the 500ms window and the
  0.05s floor-divisor case for a single just-landed sample).
