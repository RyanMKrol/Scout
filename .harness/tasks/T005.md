# T005 — Rolling throughput window (pure, transfer-time-weighted)

## Overview

The core rolling-window Mbps calculation. Scout probes in short paced bursts with idle gaps between
them, so the window must compute rate over ACTIVE TRANSFER TIME (bytes ÷ summed transfer durations),
never over wall-clock time — otherwise the pacing gaps would drag the reading toward zero. Pure struct,
instant-based, fully deterministic in tests.

## Do

Create `Sources/Domain/ThroughputWindow.swift`:

```swift
struct ThroughputWindow: Sendable {
    init(window: Duration = .seconds(2))
    mutating func record(byteCount: Int, transferDuration: Duration, endedAt: ContinuousClock.Instant)
    mutating func megabitsPerSecond(at now: ContinuousClock.Instant) -> Double?
}
```

Semantics:

- `record` appends a sample. Ignore (drop silently) samples with `byteCount <= 0` or
  `transferDuration <= .zero`.
- `megabitsPerSecond(at:)` first evicts samples whose `endedAt` is older than `now - window`, then:
  - no samples remaining → return `nil` (caller decides what "no data" means);
  - else return `(Σ byteCount × 8) / (Σ transferDuration in seconds) / 1_000_000` — i.e. bits over
    ACTIVE transfer seconds only. Idle time between probes MUST NOT dilute the rate.
- Eviction also happens in `record` so memory stays bounded.
- No clock is owned: the type only compares `ContinuousClock.Instant` values handed to it. Never call
  `ContinuousClock().now`, `Date()`, or any timer inside the type — this is a repo-wide determinism
  rule (injected time only).

## Done when

- A test file (e.g. `Tests/ThroughputWindowTests.swift`) constructs instants from a single base
  (`let base = ContinuousClock().now` then `base.advanced(by:)` — instants may be fabricated this way)
  and asserts at least:
  - one sample of 250_000 bytes over 0.2 s → 10.0 Mbps (±0.001);
  - two samples (250_000 B/0.2 s and 125_000 B/0.2 s) → 7.5 Mbps (weighted by transfer time);
  - a sample recorded 3 s before `now` with `window == 2 s` is evicted → reading reflects only the
    fresh sample; all samples stale → `nil`;
  - `byteCount: 0` and `transferDuration: .zero` samples are ignored;
  - a paced pattern — 4 samples each 256_000 B / 0.1 s, spaced 0.5 s apart — reads ~20.48 Mbps
    (active-time weighting), NOT ~4 Mbps (wall-clock dilution).
- All tests pass in the standard DoD run.
