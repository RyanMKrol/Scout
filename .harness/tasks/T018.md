# T018 — Incremental HTTP/1.1 response parser (pure)

## Overview

The meaty pure piece of the live measurement engine: an incremental parser that consumes HTTP/1.1
response bytes arriving in arbitrary chunks on a persistent keep-alive connection, tracks body
progress against Content-Length, and resets for the next response. Fully unit-tested; zero
networking in this task.

## Do

Create `Sources/Live/HTTPResponseParser.swift`:

```swift
struct HTTPResponseParser {
    enum Event: Equatable {
        case needMoreData
        case headersComplete(statusCode: Int, contentLength: Int?)
        case bodyProgress(newBodyBytes: Int)     // body bytes in the chunk just fed
        case responseComplete                    // current response fully consumed; parser resets
    }
    enum ParserError: Error, Equatable { case malformedResponse, nonSuccessStatus(Int) }

    mutating func feed(_ data: Data) throws -> [Event]
}
```

Semantics:

- Parse the status line (`HTTP/1.1 200 OK` → 200). A status outside 2xx →
  `throw ParserError.nonSuccessStatus(code)` (the sampler treats it as a failed probe — Cloudflare
  403s oversized requests, and measuring an error body as throughput would corrupt readings).
- Headers end at `\r\n\r\n`; extract `Content-Length` case-insensitively. Missing Content-Length →
  `malformedResponse` (both Cloudflare endpoints send it; Scout doesn't need chunked encoding).
- Body: count bytes until Content-Length is consumed → emit `responseComplete` and RESET so
  subsequent bytes belong to the next response (keep-alive pipelining of sequential probes).
- One `feed` call may produce several events (e.g. headers + body + complete in a single chunk),
  and leftover bytes after `responseComplete` must roll into the next response's parsing.

## Done when

- Tests (e.g. `Tests/HTTPResponseParserTests.swift`) cover at least:
  - a full 200 response fed in ONE chunk → `[headersComplete(200, N), bodyProgress(N),
    responseComplete]`;
  - the same response fed byte-by-byte (loop) accumulates the same body total and ends with
    `responseComplete`;
  - a chunk boundary splitting the `\r\n\r\n` and one splitting the status line both parse
    correctly;
  - two back-to-back keep-alive responses in one buffer → two `responseComplete`s with correct
    per-response body counts;
  - `403` status → throws `.nonSuccessStatus(403)`; garbage first line → `.malformedResponse`;
  - header name case-insensitivity (`content-length: 5` works).
- No sockets/URLSession/Network imports (pure Foundation); standard DoD run green.
