# T039 — Staleness decay: dead spots read as dead (stop holding a stale healthy number)

## Overview
A signal detector must go quiet where there's no signal. Today, if probes stop arriving but the
cellular path still reports "satisfied" (deep congestion, server stall), Scout **holds the last
reading on screen** — so sweeping the phone into a dead spot keeps showing a healthy number until
the connection formally drops. This is the recorded "Reading holds (doesn't decay) when probes
stall" limitation, and it directly undermines the live-hunting use case. This task makes a stalled
reading actively fall toward zero and surfaces a "measuring…" state.

## Do
In `Sources/Domain/SweepSession.swift`, add a staleness rule: when **no fresh throughput sample has
arrived for ~1s** (threshold a single named constant, in the 0.75–1s range), the published
`downloadMbps` / `uploadMbps` must decay toward `0` (a short linear fade to 0 over ~1s is fine, or
snap to 0), and the session must expose a stalled/measuring flag (e.g. `isStalled` or a
`measurementState` enum) alongside the existing `isMeasuring` / `cellularAvailable`. This decay must
be driven off the same injected clock seam the session uses for time (never bare `Date()` / `.now`),
so it is deterministically testable. In `Sources/Screens/MeasuringView.swift`, when the session is
stalled, render a clear **"measuring…"** indication in place of a confidently-healthy number (reuse
the existing status row / identifiers where possible; give any new element a stable
`accessibilityIdentifier`).

## Done when
- `SweepSession` exposes a stalled/measuring state, and after **> ~1s** with no new sample the
  published Mbps decays toward 0 rather than holding the last value.
- `Tests/SweepSessionTests.swift` proves the behavior with the injected clock: feed samples, then
  advance the clock past the threshold with no new sample, and assert the reading decays toward 0 and
  the stalled flag is set; a fresh sample clears it.
- `MeasuringView` shows a "measuring…" state (not a stale healthy reading) when stalled, with a
  stable `accessibilityIdentifier`; the screen builds and launches on Scout-Sim.
- On device (manual, T025-style): sweeping into a genuine dead spot makes the number fall away
  quickly instead of freezing on a good value.

## Notes
- Do NOT edit `.harness/**` here. Retiring the "Reading holds (doesn't decay)…" row in the
  limitations ledger is done by the human integrator at merge (the scope gate blocks the loop from
  editing the harness tree).
