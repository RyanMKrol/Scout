# T040 — Wire up the 250ms display tick (the specified 4Hz readout), decoupled from probe completion

## Overview
The design specifies the reading updates "~4×/sec (every 250ms)". A constant for it already exists —
`ScoutTheme.tickInterval = 0.25` (`Sources/DesignSystem/ScoutTheme.swift:75`) — but it is **dead**:
referenced nowhere, and the on-screen number is repainted only when a network probe completes. This
task makes the readout re-evaluate the rolling window and republish on a **250ms timer**, independent
of when probes land, so (combined with the short window from T037 and the decay from T039) the number
eases smoothly at 4Hz instead of jumping once per probe. Depends on T037 so the tick reads from the
short, responsive window.

## Do
In `Sources/Domain/SweepSession.swift`, add a repeating main-actor tick at
`ScoutTheme.tickInterval` (0.25s) that recomputes `downloadWindow.megabitsPerSecond(at: now)` and
`uploadWindow.megabitsPerSecond(at: now)` from the current time and republishes `downloadMbps` /
`uploadMbps` (applying the same caps as `record(_:)` at `SweepSession.swift:116`). The tick's `now`
must come from the session's injected clock seam (never bare `.now`), so the cadence is deterministic
in tests; start it in `start()` / `subscribeToSamples()` and tear it down in `stop()` alongside the
existing `sampleTask` / `radioTask` / `pathTask`. Reference `ScoutTheme.tickInterval` so the constant
is no longer dead. Probe-driven `record(_:)` still ingests samples into the windows; the tick only
governs how often the *published, displayed* value is refreshed.

## Done when
- `ScoutTheme.tickInterval` is referenced by `SweepSession` (no longer a dead constant), and the
  displayed `downloadMbps` / `uploadMbps` are recomputed from the rolling window and republished on a
  ~0.25s cadence independent of probe arrival.
- The tick is torn down in `stop()` (no leaked `Task`/timer after the session stops) and runs on the
  main actor.
- `Tests/SweepSessionTests.swift` proves, with the injected clock: with samples resident in the
  window and no new probe arriving, advancing the clock by the tick interval recomputes and
  republishes the value (i.e. the display refreshes between probes), and the tick stops after
  `stop()`.
- On device (manual, T025-style): while measuring, the hero number updates smoothly ~4×/sec rather
  than snapping once per probe.
