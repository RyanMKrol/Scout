# T038 — Download-biased, tighter probing (fresh hero data 3–4×/sec) + honest data-budget copy

## Overview
The big central number is the **download** figure — that's what you watch while hunting. Today the
sampler strictly alternates download/upload (`Sources/Live/CellularThroughputSampler.swift:35`) and
paces probes ≥500ms apart (`Sources/Live/ProbePacer.swift:4`, `interval = .milliseconds(500)`), so
the download reading only gets fresh data about **once per second**. This task biases probing toward
download and tightens the pacing so the hero metric refreshes ~3–4×/sec. Aggressive-mode decision is
locked: this **increases cellular data use** — likely past the current "10–40 MB/min" budget — so
the user-facing consent/README copy that promises that figure MUST be corrected in the same change,
or the app makes a claim it no longer honors.

## Do
1. In `Sources/Live/ProbePacer.swift`, reduce `interval` from `.milliseconds(500)` to
   **`.milliseconds(250)`** (keep it a single named constant; the `delayBeforeNextProbe` pacing logic
   is unchanged).
2. In `Sources/Live/CellularThroughputSampler.swift`, replace the strict 50/50 alternation at
   line ~35 (`direction = direction == .download ? .upload : .download`) with a **download-biased**
   schedule — default **3 download probes per 1 upload probe** (e.g. every 4th probe is upload). Keep
   the download/upload probe bodies themselves unchanged; only the direction *sequence* changes.
3. Correct the data-budget copy everywhere it appears, since the honest figure is now higher.
   Recompute from `probeBytes` (262144 B ≈ 256 KB) × the new probe rate and state a truthful range
   (on the order of **~30–60 MB/min** sustained — verify the arithmetic; do not keep the old
   "10–40" number). Update:
   - `Sources/Screens/ConsentView.swift:40` (the `bold: "10–40 MB per minute"` string), and
   - every data-budget reference in `README.md` (lines ~50, 73, 99, 125, 167 — grep
     `10.40 MB` / `per minute`).
