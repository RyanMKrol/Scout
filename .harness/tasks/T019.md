# T019 — Live cellular throughput sampler (NetworkConnection, forced .cellular)

## Overview

The real measurement engine: a `ThroughputSampling` implementation that opens a TLS connection PINNED
to the cellular interface (iOS 26 `NetworkConnection` — the only public way to force cellular; Wi-Fi
being connected must not matter), alternates paced 256 KiB download/upload probes against Cloudflare's
speed endpoints, times them, and yields `ThroughputSample`s. Cancellation tears down the connection
immediately (P0). CI cannot exercise a live cellular transfer — correctness here is: exact API usage
as specced, pure pieces already tested (T018), and the human device gate at the end of the backlog.

## Do

Create `Sources/Live/CellularThroughputSampler.swift`:

```swift
final class CellularThroughputSampler: ThroughputSampling { init() }
```

1. **Connection** (exact iOS 26 API — this snippet is SDK-typecheck-verified; use it as written):

```swift
import Network

let connection = NetworkConnection(
    to: .hostPort(host: "speed.cloudflare.com", port: 443),
    using: .parameters { TLS() }                 // TLS infers TCP + IP; SNI from the hostname
        .requiredInterfaceType(.cellular)        // HARD pin: cellular or nothing
        .multipathServiceType(.disabled)
)
```

   Notes that MUST be respected:
   - There is NO `cancel()` on `NetworkConnection` — teardown is Task-cancellation + releasing the
     reference. A pending `receive` aborts when the awaiting Task is cancelled.
   - When cellular is unavailable the connection sits in `.waiting` and `send`/`receive` SUSPEND
     (they do not fail). Do NOT add your own path monitoring here — the session already pauses the
     sampler via `CellularPathMonitor`; but DO bound the first handshake so a stuck probe loop can't
     hang forever: use `TCP()` builder's `connectionTimeout` if you include a TCP options step, or
     rely on cancellation (preferred: rely on cancellation; keep it simple).
   - Any `switch` over connection state enums needs `@unknown default` under Swift 6.

2. **Probe loop** — inside the `AsyncStream`'s owned `Task` (`onTermination` cancels it):
   - Alternate: download probe, upload probe, download, … paced by
     `ProbePacer.delayBeforeNextProbe` (sleep via `ContinuousClock` — cancellable).
   - Download probe: `try await connection.send(HTTPProbeRequest.download())`; record
     `start = ContinuousClock().now`; loop `let msg = try await connection.receive(atLeast: 1,
     atMost: 262_144)` feeding `msg.content` into an `HTTPResponseParser` until `responseComplete`;
     `endedAt = .now`; yield `ThroughputSample(direction: .download, byteCount: <Content-Length>,
     transferDuration: endedAt - start, endedAt: endedAt)`.
   - Upload probe: `start = .now`; `send(uploadHeader())` then `send(uploadBody())`; then receive
     until the response's `responseComplete` (its body is small); `endedAt = .now`; yield
     `.upload` sample with `byteCount: HTTPProbeRequest.probeBytes` and
     `transferDuration: endedAt - start`. (Duration includes the server acknowledging the POST —
     a slight underestimate at high speed; acceptable for a capped 5 Mbps instrument.)
   - Errors (thrown receive/send, parser error, non-2xx): drop the connection, sleep 1 s
     (cancellable), create a FRESH connection, continue the loop. Never crash the stream on a probe
     failure; never yield a sample for a failed probe.
   - Header bytes are excluded from `byteCount` by construction (Content-Length only).
3. **Concurrency**: the class is `Sendable` (immutable config; state lives inside the stream's Task).
   `samples()` is single-subscriber (documented on the protocol).
4. **Wire the composition root**: the DEVICE branch (no `-ScoutScenario` argument) now uses
   `CellularThroughputSampler()`; simulator and forced-scenario launches keep `SimulatedSampler`.
   This completes `AppEnvironment`'s live wiring (radio + path + sampler).
