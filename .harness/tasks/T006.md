# T006 — Measurement seams: protocols + simulated (scripted) providers

## Overview

The protocol seams every measurement source implements, plus the SIMULATED implementations Scout uses
in the iOS Simulator (which has no cellular radio), in UI tests, and in the harness's screenshot
checks. Live cellular/radio I/O can never run in CI, so these seams + fakes are what make the whole
app testable — they are a frozen contract that later tasks (session engine, live providers, app shell)
bind to verbatim.

## Do

1. Create `Sources/Domain/MeasurementSeams.swift` with EXACTLY these shapes:

```swift
enum TransferDirection: Sendable, Equatable {
    case download, upload
}

struct ThroughputSample: Sendable, Equatable {
    let direction: TransferDirection
    let byteCount: Int                       // payload bytes moved (HTTP headers excluded)
    let transferDuration: Duration           // active transfer time for those bytes
    let endedAt: ContinuousClock.Instant
}

protocol ThroughputSampling: Sendable {
    /// Single-subscriber stream of probe results. Finishing/cancelling the consuming task MUST
    /// tear down all underlying transfer activity (P0: no data moves once the consumer stops).
    func samples() -> AsyncStream<ThroughputSample>
}

protocol RadioInfoProviding: Sendable {
    /// Yields the current generation immediately on subscription, then again on every change.
    func generations() -> AsyncStream<RadioGeneration>
}

protocol CellularPathMonitoring: Sendable {
    /// Yields current cellular availability immediately on subscription, then on every change.
    func availability() -> AsyncStream<Bool>
}
```

2. `Sources/Domain/Simulation.swift` — the scripted implementations. The random-walk math is a PURE,
   seeded-RNG-driven function so it is unit-testable; the streams are thin timing wrappers around it.

```swift
enum SimulationScenario: String, Sendable, Equatable {
    case live = "Live", great = "Great", usable = "Usable", poor = "Poor", none = "None"
}

enum ScenarioWalk {
    static func initialDownloadMbps(_ scenario: SimulationScenario) -> Double
    static func nextDownloadMbps(previous: Double, scenario: SimulationScenario,
                                 using rng: inout some RandomNumberGenerator) -> Double
    static func uploadMbps(forDownload down: Double,
                           using rng: inout some RandomNumberGenerator) -> Double
}

final class SimulatedSampler: ThroughputSampling { init(scenario: SimulationScenario) }
final class SimulatedRadioProvider: RadioInfoProviding { init(scenario: SimulationScenario) }
final class SimulatedPathMonitor: CellularPathMonitoring { init(scenario: SimulationScenario) }
```

   Walk math (adapted from the design prototype's simulation in
   `design/design_handoff_scout/Scout.dc.html`, rescaled to Scout's capped 0–10 Mbps instrument):
   - Bands (download): great `[6.5, 9.8]`, usable `[2.4, 5.5]`, poor `[0.3, 1.4]`.
   - Initial values: great 8.15, usable 3.95, poor 0.85, live 4.0, none 0.
   - Banded scenarios: `next = prev + (center − prev) × 0.28 + (rand01 − 0.5) × (hi − lo) × 0.4`,
     then clamp to `[lo × 0.7, min(hi × 1.12, 10.0)]` (rand01 = uniform Double in 0..<1 from the rng).
   - `.live`: `next = prev + (rand01 − 0.5) × 1.3`; with probability 0.06 add `(rand01 − 0.4) × 4.3`;
     clamp to `[0.2, 9.9]`.
   - Upload: `down × 0.45 + (rand01 − 0.5) × 0.6`, clamped to `[0.1, 5.0]`.
   - `.none`: walk irrelevant (sampler emits nothing).
   - `SimulatedSampler`: every 250 ms emit ONE download sample then ONE upload sample. Fabricate probe
     shape: `transferDuration = .milliseconds(180)`, `byteCount = Int(mbps × 1_000_000 × 0.18 / 8)`,
     `endedAt = ContinuousClock().now` at emission (real now is fine here — this is the live-timing
     wrapper, not the pure math). Sleep via `try? await ContinuousClock().sleep(...)` inside a `Task`
     owned by the stream; the stream's `onTermination` MUST cancel that task. Scenario `.none` → yield
     nothing (stream stays open, no samples).
   - `SimulatedRadioProvider`: immediately yield — great: `.fiveG`; usable: `.fiveG`; poor: `.lte`;
     live: `.fiveG`; none: `.unknown`. (Static per scenario is fine for v1.)
   - `SimulatedPathMonitor`: immediately yield `false` for `.none`, `true` otherwise; then keep the
     stream open.

## Done when

- All shapes above compile EXACTLY as written (names, signatures) — later tasks reference them verbatim.
- A test file (e.g. `Tests/ScenarioWalkTests.swift`) with a seeded RNG (e.g. a tiny SplitMix64
  `RandomNumberGenerator` defined in the test target) asserts:
  - for each banded scenario, iterating `nextDownloadMbps` 200× from the initial value never leaves
    the clamp range, and all `.great` values stay ≥ 4.55 (6.5 × 0.7) — i.e. band character holds;
  - `.live` values stay within [0.2, 9.9] over 200 iterations;
  - `uploadMbps` stays within [0.1, 5.0] and correlates with its download input (down 8 → up > down 1's up, on average over 50 draws);
  - `SimulatedSampler` for `.great`: collecting the first 4 samples from `samples()` yields
    alternating directions starting with `.download`, positive byte counts, and 180 ms durations
    (a short real await in the test with a bounded timeout is acceptable for this one integration
    assertion; everything else is pure).
- Tests pass in the standard DoD run. No live networking anywhere in this task.
