## Overview

Create the Scout Xcode project and turn the Definition-of-Done tooling on, so CI goes green on an empty
build. This is the ONE task that defines the project-wide conventions (SwiftFormat, SwiftLint, the Xcode
scheme, CI on a macOS runner) that gate every later build вАФ and creating an `.xcodeproj` is fiddly,
GUI-driven, one-time work with no automation payoff вАФ so a human sets it up once, by hand, rather than the
loop. (`gate: needs-human`: the loop skips this task and records `failed:blocked` until you mark it done.)

## Do

1. **Create the Xcode project.** In Xcode вЖТ *File вЦЄ New вЦЄ Project вЦЄ iOS вЦЄ App*:
   - Product name **Scout**, interface **SwiftUI**, language **Swift**, storage **None**.
   - Set the **minimum deployment target to iOS 26** (per the README's stack decision).
   - Save it at the repo root so the `.xcodeproj` and the `Scout/` sources live under
     `/Users/ryankrol/Development/Scout`.
   - Confirm the scheme is named **`Scout`** and is **shared** (Xcode вЦЄ Manage Schemes вЦЄ вЬУ Shared) so
     CI and the loop can `xcodebuild -scheme Scout`. The default `ContentView` + a trivial test target
     are enough for an "empty build".
2. **Add the DoD tooling configs** at the repo root:
   - **`.swiftformat`** вАФ a minimal SwiftFormat config (rules/version pin) so `swiftformat --lint .`
     is stable.
   - **`.swiftlint.yml`** вАФ a minimal SwiftLint config. Keep it lenient at first so `swiftlint --strict`
     passes on the generated template; tighten later.
3. **Confirm CI matches the DoD.** `.github/workflows/ci.yml` already runs the real commands on a macOS
   runner (`swiftformat --lint .`, `swiftlint --strict`, `xcodebuild test -scheme Scout вА¶`). Verify the
   simulator name in the `-destination` (and in `.harness/config/harness.env`'s `LOCAL_DOD`) matches one
   the runner actually has вАФ check the CI log's `Show toolchain` step and `xcrun simctl list devices`
   locally. **Note:** iOS 26 needs **Xcode 26** on the runner; if `macos-latest` lags, pin a newer runner
   image or `sudo xcode-select -s` a specific Xcode in the workflow.
4. **Get it green** вАФ `swiftformat --lint .`, `swiftlint --strict`, and the `xcodebuild test` command all
   pass locally, then CI passes on `main`.

## Done when

CI is green on `main`; the DoD commands CI runs match `.harness/config/harness.env`'s `LOCAL_DOD` exactly
(SwiftFormat + SwiftLint + `xcodebuild test -scheme Scout`); the `Scout` scheme is shared; and a local
`swiftformat --lint .` / `swiftlint --strict` run is clean on the generated template. (SwiftFormat/SwiftLint
only scan `.swift`, of which none live under `.harness/`, so no ignore file is needed.) Then mark this task
done so its dependents unblock: `.harness/scripts/mark-done.sh T001` (or the dashboard's "Mark done").
