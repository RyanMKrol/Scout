## Overview

Stand up the Scout Xcode project **via XcodeGen** and turn the Definition-of-Done tooling on, so CI goes
green on an empty build. This is the ONE task that defines the project-wide conventions (the `project.yml`
spec, SwiftFormat, SwiftLint, CI on a macOS runner) that gate every later build. It's fiddly, GUI-adjacent,
one-time work â€” so a human does it once, by hand. (`gate: needs-human`: the loop skips this task and records
`failed:blocked` until you mark it done.)

**Why XcodeGen (not a hand-built .xcodeproj):** the autonomous loop must be able to add new `.swift` files
to the build target on its own. A committed, hand-built `.xcodeproj` can only grow via the Xcode GUI â€” so
the loop would break on the first task that creates a file. With XcodeGen, `sources: [Sources]` means any
file dropped under `Sources/` is in the target on the next `xcodegen generate`. The `project.yml`,
`build_run.sh`, `.github/workflows/ci.yml`, and `.github/actions/**` are already scaffolded for you â€” this
task is mostly wiring up the source tree and confirming everything goes green.

## Do

1. **Install XcodeGen** if needed: `brew install xcodegen`.
2. **Create the minimal source tree** the committed `project.yml` expects:
   - `Sources/` with a trivial SwiftUI entrypoint â€” a `@main` `App`, a `ContentView`, and an
     `Sources/Info.plist` (XcodeGen will merge in the `project.yml` `info.properties`, incl.
     `UILaunchScreen`). Keep it to a "Hello, Scout" screen â€” just enough to build and screenshot.
   - `Tests/` with one trivial passing XCTest so the `ScoutTests` target compiles.
3. **Generate + open**: `xcodegen generate` â†’ this writes the git-ignored `Scout.xcodeproj`. Verify the
   `Scout` scheme builds and runs in the simulator.
4. **Add the DoD tooling configs** at the repo root:
   - **`.swiftformat`** â€” a minimal SwiftFormat config (rules/version pin) so `swiftformat Sources Tests --lint` is stable and green on the template.
   - **`.swiftlint.yml`** â€” a minimal SwiftLint config; keep it lenient at first so `swiftlint --strict` passes, then tighten later.
5. **Create the dedicated local simulator** the loop's visual check + local DoD use:
   `xcrun simctl create "Scout-Sim" com.apple.CoreSimulator.SimDeviceType.iPhone-16 <an installed iOS-26 runtime id>`
   (list options: `xcrun simctl list runtimes devicetypes`). `build_run.sh` defaults to this device by name.
   For local `xcodebuild test`, prefer overriding the destination to `id=<Scout-Sim UDID>` so this loop never
   collides with another iOS loop on the same Mac (see root `CLAUDE.md` â†’ Tooling notes). CI stays by-name.
6. **Confirm the whole DoD is green** locally, exactly as CI runs it:
   `xcodegen generate && swiftformat Sources Tests --lint && swiftlint --strict && xcodebuild test -project Scout.xcodeproj -scheme Scout -destination 'platform=iOS Simulator,name=iPhone 16' CODE_SIGNING_ALLOWED=NO`
   Then run `./build_run.sh` and open `screenshots/latest.png` â€” the app must **fill the whole screen**
   (no black letterbox bars; if you see them, `UILaunchScreen` isn't reaching the built Info.plist).
7. **Get CI green on `main`.** The workflow selects Xcode 26, installs XcodeGen, generates, formats, lints,
   resolves a simulator, and tests. **Note:** iOS 26 needs Xcode 26 on the runner â€” the `select-xcode`
   action fails loudly if `macos-latest` doesn't have it yet; if so, pin a newer runner image.

## Done when

CI is green on `main`; the DoD commands CI runs match `.harness/config/harness.env`'s `LOCAL_DOD` exactly
(xcodegen â†’ SwiftFormat â†’ SwiftLint â†’ `xcodebuild test`); the `Scout` scheme builds; a local run of the full
DoD command above is clean; and `./build_run.sh` produces a full-screen (non-letterboxed) `screenshots/latest.png`.
(SwiftFormat/SwiftLint only scan `.swift`, of which none live under `.harness/`, so no ignore file is needed.)
Then mark this task done so its dependents unblock: `.harness/scripts/mark-done.sh T001` (or the dashboard's
"Mark done").
