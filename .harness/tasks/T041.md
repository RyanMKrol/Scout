## Overview
Add a user-controllable pause/resume to SweepSession so the live sweep can be frozen mid-hunt and resumed, distinct from start()/stop() and from the no-consent/no-cellular empty state.

## Do
Add a `paused` state and `pause()` / `resume()` (or a `togglePause()`) to SweepSession, plus an observable `isPaused` flag. DEFAULT SEMANTICS (confirm via the relayed freeze-vs-suspend question — SUSPEND is the recommendation): on pause, stop consuming cellular data by cancelling the active sample subscription (leave the last downloadMbps/uploadMbps/quality/generation values in place as a frozen snapshot rather than zeroing them, so the UI shows the held reading); on resume, re-subscribe to samples exactly as start() does when measuring and cellular is available. Ensure pause/resume interacts correctly with existing start()/stop(), the scenePhase background->stop() path, and the cellularAvailable path-update logic (a pause should not be silently undone by a path update). Do NOT touch UI files.

## Done when
SweepSession exposes an observable `isPaused` and pause/resume entry point(s); pausing leaves the last-displayed download/upload/quality/generation values intact (does not zero them) while halting further sample consumption; resuming restores live updating; and a new XCTest in Tests/SweepSessionTests.swift (using the existing FakeThroughputSampler/FakeRadioInfoProvider/FakeCellularPathMonitor) asserts that after start()+pause() no further yielded samples change downloadMbps (frozen), the sampler subscription is torn down, and after resume() a newly yielded sample updates downloadMbps again. If the owner answers FREEZE-ONLY instead of SUSPEND, keep the sampler running on pause and gate only the published values — the test then asserts the displayed value stops changing while sessionDownloadBytes keeps climbing.
