# T037 — Short recency-weighted throughput window (live responsiveness)

## Overview
Scout is meant to be a live "signal hunter" — you sweep the phone around a room and the download
number should respond in real time as you move. Today the reading is averaged over a **2-second**
rolling window, so when you move the phone it takes up to ~2s for the number to reflect the new
spot. This task shortens that window (or switches to a recency-weighted average) so the reading
tracks the current location quickly. This is the single highest-impact change for "feels live as I
move." Aggressive-mode decision is locked: bias hard toward responsiveness, accepting a jumpier
number.

## Do
In `Sources/Domain/ThroughputWindow.swift` (and its two construction sites in
`Sources/Domain/SweepSession.swift`, currently `ThroughputWindow(window: .seconds(2))` at
`SweepSession.swift:20-21` and again in `subscribeToSamples()` around `SweepSession.swift:94-95`),
replace the 2-second averaging with a **short recency-weighted** reading. Default approach: a fixed
short window of **~0.6s** (must be ≤ 0.75s), keeping the existing transfer-time-weighted formula
(`(Σ byteCount × 8) / (Σ transferDuration seconds) / 1_000_000` over the samples still inside the
shorter window). An exponential-moving-average variant (half-life ≈ 0.4–0.6s) that weights the
newest samples is an acceptable alternative if it reads more smoothly — but it must still ignore
idle gaps between probes (never dilute the rate by wall-clock idle time), exactly as the current
window does. The chosen window duration must be a single named constant, not a magic literal.

## Done when
- The rolling window used for the live reading is **≤ 0.75s** (default ~0.6s), replacing the previous
  2-second window at every construction site in `SweepSession` — no `.seconds(2)` window remains for
  the live download/upload reading.
- The transfer-time-weighted math is preserved: the rate is bits over **active transfer seconds**,
  never diluted by idle time between probes; samples with `byteCount <= 0` or
  `transferDuration <= .zero` are still dropped.
- `Tests/ThroughputWindowTests.swift` proves, with the injected `ContinuousClock`-style clock the
  suite already uses (never bare `.now`): (a) a sample older than the new short window is evicted;
  (b) after a step change in throughput, the reported Mbps converges to the new level within roughly
  one short-window span rather than ~2s.
- On device (manual, T025-style): sweeping the phone between a strong and a weak spot moves the hero
  download number within about a second, not after a multi-second lag.
