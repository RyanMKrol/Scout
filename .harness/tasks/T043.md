## Overview
Enabling half of the re-attempt of T042. T042 (tap the dial center to pause/resume the live sweep) was blocked at the first build for scope-creep, but a deeper blocker is that its required XCUITest can't reach the LIVE measuring state: the only existing launch-arg seam, `ScoutForceIdle`, deliberately forces consent OFF and lands MeasuringView on the no-consent paused empty state (RootView sets `consentGiven = false` when forceIdle is set). There is NO seam that starts the app consent-given with a running, cellular-available session — the exact state a UI test must be in to see and tap the dial. This unit adds that seam so unit B's XCUITest is actually runnable in the harness.

## Do
Add a NEW launch-argument seam, `-ScoutForceMeasuring` (read via `UserDefaults.standard.bool(forKey: "ScoutForceMeasuring")` in RootView.init, exactly like the existing `ScoutForceIdle`/`ScoutHoldSplash` keys — dash-prefixed launch args auto-populate the UserDefaults argument domain). When set: construct the AppRouter with `consentGiven: true`, skip the splash hold if appropriate, and in the `.task` splash-finished path land on `.measuring` with `router.consentGiven == true` AND call `session.start()` so the session is live (on the simulator `AppEnvironment.makeSession()` already wires a SimulatedSampler/SimulatedRadioProvider/SimulatedPathMonitor, so `session.cellularAvailable` becomes true and readings update — no live cellular needed). The pass-through `consentGiven:` given to MeasuringView must be `true` in this mode (contrast: `ScoutForceIdle` passes `router.consentGiven && !forceIdle`, i.e. false — do NOT copy that here). Net effect: launching with `-ScoutForceMeasuring 1` shows `measuringContent()` (the live dial), not `pausedContent()`. Do NOT change the default (no-arg) behaviour, and do NOT alter or remove the existing `ScoutForceIdle` seam — both must keep working independently. Consider having `-ScoutScenario` still honoured alongside it (a test can pass `-ScoutScenario greatSignal -ScoutForceMeasuring 1`).

## Done when
Launching the app with the `-ScoutForceMeasuring 1` launch argument (optionally with `-ScoutScenario <name>`) starts directly in the live measuring state: MeasuringView renders `measuringContent()` with a running session (`session.cellularAvailable == true`, readings ticking), NOT the no-consent paused empty state; the existing `-ScoutForceIdle` behaviour and the default no-argument launch flow are both unchanged. No new production behaviour ships when neither arg is present.
