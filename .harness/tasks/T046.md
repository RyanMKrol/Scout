# T046 — Wall-clock throughput window (bytes-in-last-0.5s ÷ 0.5s)

## Overview
With a continuous download (T045), the right speed reading is "how many bytes arrived in the last
half-second, divided by half a second" — a true wall-clock rate. Today `ThroughputWindow`
(`Sources/Domain/ThroughputWindow.swift`) instead divides bytes by the *summed active transfer
duration* (`megabitsPerSecond` = `totalBits / Σ transferDuration`), which for a fast tiny probe returns
its burst rate and pegs the cap. This task replaces that math with a wall-clock sliding window.
**This supersedes T037's transfer-time-weighted window.**

## Do
In `Sources/Domain/ThroughputWindow.swift`, change the reading to a wall-clock rate over a fixed
recency window: keep each recorded sample's `byteCount` and its arrival instant (`endedAt`), evict
samples older than the window, and compute `megabitsPerSecond(at: now)` as
`(Σ byteCount in window × 8) / windowSeconds / 1_000_000` — i.e. bytes over ELAPSED wall-clock seconds,
NOT over summed transfer durations. Set the default `liveWindow` to **0.5s** (a single named constant;
replaces the current 600ms). Use the actual covered span when the window isn't yet full (divide by
`min(windowSeconds, now − oldestSampleInstant)` clamped to a small positive floor) so a just-started
stream isn't under-reported, but once warmed the divisor is the full window. The `transferDuration`
parameter to `record(...)` becomes vestigial — keep the signature for compatibility but do not use it
in the rate; still drop samples with `byteCount <= 0`.

## Done when
- `ThroughputWindow.megabitsPerSecond(at:)` returns a wall-clock rate: bytes in the trailing window × 8
  ÷ elapsed seconds ÷ 1e6, with the default window = 0.5s (named constant); no dependence on
  `transferDuration` remains in the computed rate.
- `Tests/ThroughputWindowTests.swift` proves, with the injected `ContinuousClock`-style clock: (a) a
  steady arrival of B bytes every Δt reports ≈ `B/Δt` converted to Mbps regardless of per-chunk
  duration; (b) samples older than 0.5s are evicted so the rate reflects only recent arrivals; (c) when
  arrivals stop, advancing `now` past the window drives the rate to 0/nil (not a held value).
- The math is unit-correct: e.g. 1,000,000 bytes arriving over the last 0.5s reads ≈ 16 Mbps.
