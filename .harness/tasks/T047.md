# T047 — Wire continuous samples through SweepSession (live reading off the wall-clock window)

## Overview
Connect the continuous sampler (T045) and wall-clock window (T046) so the hero download number is a
live, smoothly-updating rate driven by the 250ms display tick — not a per-probe spike. Today
`SweepSession.record(_:)` (`Sources/Domain/SweepSession.swift`, ~line 199) recomputes Mbps at each
sample's `endedAt`, and the staleness path (`isStalled` / `lastSampleAt`) currently fires between the
old discrete probes. With continuous flow the window is populated every tick, and staleness should mean
the stream genuinely stalled (a real dead spot).

## Do
Update `SweepSession` so that: (1) each incoming continuous `ThroughputSample` is recorded into the
wall-clock `downloadWindow` (and accumulated into `sessionDownloadBytes`); (2) the published
`downloadMbps` is recomputed from `downloadWindow.megabitsPerSecond(at: now())` on the existing 250ms
display tick (`tickTask`, `ScoutMotion.tickInterval`) using the injected clock's `now()`, so it updates
smoothly between chunk arrivals rather than only on `record`; (3) the staleness logic
(`stalenessTask` / `isStalled` / the decay) now triggers only when NO chunk has arrived for the stall
threshold — i.e. the continuous stream actually stopped — giving a true "dead spot" signal instead of
firing every inter-probe gap. Keep the display-cap application exactly as it is for now
(`min(value, ScoutMeter.downloadCapMbps)`) — uncapping is T049's job. Do not touch UI files.

## Done when
- `downloadMbps` is recomputed from the wall-clock window on the 250ms tick (via the injected clock),
  and reflects the continuously-arriving chunks; `sessionDownloadBytes` climbs with the stream.
- The staleness/stall signal (`isStalled`) fires only after a genuine gap in chunk arrivals (stream
  stopped), not between normal continuous reads.
- `Tests/SweepSessionTests.swift` proves, with the fake sampler + injected clock: feeding a steady
  continuous chunk stream yields a stable non-zero `downloadMbps` recomputed on ticks (not collapsing
  between chunks); stopping the stream and advancing the clock past the stall threshold sets `isStalled`
  and decays the reading toward 0.
- On device (manual, T025-style): the hero number holds a live value and tracks the connection as you
  sweep, instead of spiking to the cap then falling to zero.
